1.  Type name
------------------------------------------------------------------------

    application

------------------------------------------------------------------------
2.  Subtype name
------------------------------------------------------------------------

    vnd.k9+nickel

    The "+nickel" structured syntax suffix indicates that the format's
    validation layer is expressed in the Nickel configuration language
    (https://nickel-lang.org).

------------------------------------------------------------------------
3.  Required parameters
------------------------------------------------------------------------

    N/A

------------------------------------------------------------------------
4.  Optional parameters
------------------------------------------------------------------------

    security-level
        One of "kennel", "yard", or "hunt".  Indicates the maximum
        execution privilege the component requests.  Defaults to
        "kennel" if absent.  See Section 6 for security implications
        of each level.

    version
        The K9 specification version the component conforms to.
        Syntax: major.minor.patch (e.g., "1.0.0").

------------------------------------------------------------------------
5.  Encoding considerations
------------------------------------------------------------------------

    8-bit text (with binary note)

    K9 files begin with the ASCII magic number "K9!" (0x4B 0x39 0x21)
    followed by UTF-8 text.  The textual body uses YAML-like syntax
    for the data layer and Nickel for the contract layer.  Opaque
    payloads embedded via the pedigree system may contain arbitrary
    binary data.  Line endings are LF (U+000A) by convention; parsers
    MUST accept CR+LF (U+000D U+000A) as well.

------------------------------------------------------------------------
6.  Security considerations
------------------------------------------------------------------------

    K9 implements a tiered security model called the Leash System.
    The security implications differ significantly by level:

    Kennel level (Pure Data):
        No execution.  Read-only.  The file contains only structured
        data (metadata, content blocks, tags).  Processors MUST NOT
        execute any content.  Safe to open in any environment,
        including constrained devices (ASIC, Edge nodes, Minix).

    Yard level (Validation Only):
        Nickel contract evaluation is permitted.  Nickel is a
        functionally pure language with no side effects, no filesystem
        access, and no network access.  Evaluation verifies that the
        component's data satisfies its declared type contracts.
        Processors MUST evaluate Nickel contracts in a sandboxed
        context.  No external resource access is permitted at this
        level.

    Hunt level (Full Execution):
        Complete must-just-nickel triad execution is permitted.  Just
        recipes MAY perform filesystem operations, invoke external
        commands, and deploy components.  This level REQUIRES:

          - A valid Ed25519 cryptographic signature over the
            component's pedigree and recipes
          - Explicit user consent before execution
          - Podman-first containerised execution where available
          - Fail-fast on signature verification failure

        Unsigned or improperly signed Hunt-level components MUST be
        rejected without execution.

    Threat mitigations:

      Malicious payload:  Crafted .k9 files with hostile Just recipes
      are mitigated by the Hunt-level signature requirement.  Kennel
      and Yard levels cannot execute recipes.

      Contract isolation:  Even if a Just recipe is compromised, it
      can only act on resources explicitly granted by the Nickel
      contracts.

      Complexity exhaustion:  Nickel evaluation is bounded.  Kennel
      mode is available for environments where any evaluation is too
      expensive.

      Dependability collapse prevention:  The tiered model ensures
      that security controls scale with the component's requested
      privileges, preventing the common failure mode where excessive
      security causes users to bypass all controls.

    Privacy:  K9 pedigree metadata may contain author information,
    version history, and deployment targets.  Implementations SHOULD
    provide mechanisms to strip metadata before sharing.

------------------------------------------------------------------------
7.  Interoperability considerations
------------------------------------------------------------------------

    K9 is designed for multi-architecture permanence.  Components are
    intended to be portable across:

      - Linux (all architectures)
      - Minix
      - macOS
      - Android
      - Embedded systems and ASICs (Kennel level only)

    The must-just-nickel triad provides environment detection (must
    script), task orchestration (Just recipes), and typed validation
    (Nickel contracts).  The "must" bootstrap script detects the host
    environment and ensures prerequisites are available.

    Interoperability at the data level (Kennel) requires only a YAML
    parser and the K9! magic number detector.  Yard-level
    interoperability requires a Nickel evaluator.  Hunt-level
    interoperability requires Nickel, Just, and Ed25519 signature
    verification.

    The K9 format is self-describing: conforming repositories include
    a pedigree.ncl file that declares the component's breed, version,
    security level, and target architectures.

10.  Fragment identifier considerations
------------------------------------------------------------------------

    Fragment identifiers for K9 documents use dot-separated paths into
    the YAML/Nickel data structure.

    Syntax:  #<path> where <path> is a dot-separated key sequence
    (e.g., "metadata.name", "content.greeting")

    Resolution:  Fragment MUST resolve to a key in the component's
    data structure.  If no match, the user agent SHOULD ignore the
    fragment.

Other comments
------------------------------------------------------------------------

    K9 (Self-Validating Components) treats files as active, self-
    validating entities rather than passive data containers.  The
    must-just-nickel triad enables environment-aware deployment:
    environment detection via POSIX shell (must), task orchestration
    via Just recipes, and typed validation via Nickel contracts.

    The three-tier Leash System (Kennel/Yard/Hunt) ensures that
    security controls are proportional to the component's requested
    privileges, preventing dependability collapse while maintaining
    strong guarantees for executable components.

    The K9! magic number enables immediate kernel-level file type
    identification without content inspection.
